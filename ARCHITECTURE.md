# Архитектура

## Обзор

Приложение состоит из React-клиента, реактивного Spring Boot API, PostgreSQL и локального файлового хранилища.

```text
React + TypeScript (5173)
          |
          | /api через Vite proxy
          v
Spring Boot WebFlux (8081)
          |
          +---- R2DBC ----> PostgreSQL (5433)
          |
          +--------------> data/uploads
```

## Backend

Backend использует Java 21, Spring Boot 4, Spring WebFlux и Spring Data R2DBC.

Код разделён по предметным областям:

```text
src/main/java/com/example/filestorage
├── bookmark   — закладки
├── auth       — пользователи, сессии и подтверждение email
├── folder     — папки
├── note       — заметки
├── shared     — общие ошибки и обработчик исключений
└── storage    — файлы и физическое хранилище
```

Каждая область содержит контроллер, сервис, модель, репозиторий и при необходимости DTO запроса.

### Реактивный поток

Контроллеры возвращают `Mono` или `Flux`. Репозитории работают через R2DBC без блокирующего JDBC. Создание сущностей с UUID выполняется через `R2dbcEntityTemplate.insert`, а обновление существующих сущностей — через `ReactiveCrudRepository.save`.

### API

| Область | Базовый адрес | Операции |
|---|---|---|
| Папки | `/api/folders` | список, создание, переименование, удаление |
| Файлы | `/api/files` | список, загрузка, скачивание, удаление |
| Закладки | `/api/bookmarks` | список, создание, изменение, удаление |
| Заметки | `/api/notes` | список, создание, изменение, удаление |
| Авторизация | `/api/auth` | регистрация, подтверждение email, вход, выход, смена пароля |

Для получения вложенных папок используется `GET /api/folders?parentId={id}`. Для получения файлов папки используется `GET /api/files?folderId={id}`.

## База данных

PostgreSQL запускается из `compose.yaml` и доступен приложению на порту `5433`.

Схема находится в `src/main/resources/schema.sql` и применяется при каждом запуске благодаря `spring.sql.init.mode=always`.

### Таблицы

| Таблица | Назначение |
|---|---|
| `folders` | иерархия папок через `parent_id` |
| `stored_files` | метаданные загруженных файлов |
| `bookmarks` | названия, адреса и описания ссылок |
| `notes` | заголовки, тексты, цвета и даты заметок |
| `users` | учётные записи и BCrypt-хеши паролей |
| `auth_sessions` | SHA-256-хеши сессионных токенов |
| `email_verification_tokens` | токены подтверждения email |
| `login_verification_tokens` | одноразовые токены подтверждения входа |

Папки образуют дерево через внешний ключ `folders.parent_id -> folders.id`. Файлы связаны с папкой через `stored_files.folder_id`.

## Файловое хранилище

PostgreSQL хранит только метаданные файла. Содержимое записывается в `data/uploads` под UUID-именем из поля `storage_name`, а исходное Unicode-имя хранится отдельно в `original_name` и используется при отображении и скачивании.

Поток загрузки:

```text
multipart-запрос
  -> FileController
  -> FileStorageService
  -> запись в data/uploads
  -> INSERT метаданных в stored_files
```

При ошибке сохранения метаданных сервис удаляет уже записанный физический файл. При удалении файла сначала проверяется запись, затем удаляются физический файл и метаданные.

## Frontend

Frontend находится в `frontend` и использует React 19, TypeScript и Vite.

Основная логика приложения расположена в `frontend/src/App.tsx`, стили — в `frontend/src/App.css`.

### Состояние

React хранит:

- активный раздел;
- текущий путь и содержимое папки;
- раскрытые ветки дерева;
- файлы, закладки и заметки;
- состояния модальных окон, меню действий, загрузки, уведомлений и подтверждений.

### Дерево папок

Содержимое папки загружается только при первом раскрытии ветки. Результат сохраняется в `expandedFolders`. Повторный клик удаляет ветку из состояния и скрывает дочерние элементы. Компонент рендерит дочерние папки рекурсивно.

Пункт «Открыть» меняет текущий путь. Меню `⋮` папки позволяет загрузить файл, создать вложенную папку, открыть, переименовать или удалить папку. Создание и переименование выполняются в модальных окнах.

### Заметки

Поиск заметок выполняется на клиенте по заголовку и тексту. Создание и изменение используют одно модальное окно. Допустимые цвета ограничены одинаковым набором на frontend и backend: `white`, `yellow`, `green`, `blue`, `rose`.

### Поиск

Поиск выполняется на клиенте без дополнительных API-запросов:

- в файловом разделе фильтруются файлы и папки текущего открытого каталога по имени;
- в закладках проверяются название, URL и описание, при этом сохраняется структура групп;
- в заметках проверяются заголовок и текст, при этом сохраняется структура групп.

### Визуальный слой

- Framer Motion управляет переходами и анимациями;
- Lucide React предоставляет иконки;
- Vite перенаправляет `/api` на `http://localhost:8081`;
- интерфейс адаптируется на контрольных ширинах `980px` и `620px`.

## Группы закладок и заметок

Группы реализованы общей сущностью `content_groups`. Поле `type` принимает значение `BOOKMARK` или `NOTE`, поэтому группы разных разделов не смешиваются. Таблицы `bookmarks` и `notes` связаны с группой внешним ключом `group_id`. Frontend показывает только элементы, принадлежащие группе.

API групп доступно по адресу `/api/groups`:

- `GET /api/groups?type=BOOKMARK|NOTE` — список групп выбранного типа;
- `POST /api/groups` — создание группы;
- `PATCH /api/groups/{id}` — переименование;
- `DELETE /api/groups/{id}` — удаление пустой группы.

Непустую группу удалить нельзя. Backend также запрещает связывать заметку с группой закладок и наоборот.

### Отображение групп

```text
▼ Работа · 3                                      ⋮
  ├── карточка или закладка 1
  ├── карточка или закладка 2
  └── карточка или закладка 3

▶ Покупки · 5                                     ⋮
```

- название группы является заголовком секции;
- элементы отображаются непосредственно под заголовком своей группы;
- содержимое каждой группы можно независимо раскрывать и сворачивать;
- в заголовке отображается количество элементов;
- меню `⋮` содержит добавление, переименование и удаление;
- состояния раскрытия заметок и закладок хранятся независимо;
- поиск заметок сохраняет структуру групп и показывает совпавшие карточки внутри них;
- пустая группа показывает локальное пустое состояние;
- создание внутри секции автоматически устанавливает её `groupId`;
- редактирование позволяет перенести элемент в другую группу.

## Безопасность

Spring Security WebFlux защищает API. Публичны только получение CSRF-токена, регистрация, подтверждение email и вход. Сессионный токен передаётся в `HttpOnly`, `SameSite=Strict` cookie, а в PostgreSQL хранится только его SHA-256-хеш. Пароли хешируются BCrypt. Все запросы к папкам, файлам, группам, закладкам и заметкам ограничены `owner_id` текущего пользователя.

Подтверждения регистрации и каждого входа отправляются через настраиваемый SMTP. Ссылка входа действует 15 минут. Для HTTPS необходимо установить `SECURE_COOKIE=true`.

В режиме `MAIL_ENABLED=false` подтверждения пропускаются. Это значение используется по умолчанию и позволяет запускать приложение без SMTP.

## Production-развёртывание

Рабочее развёртывание на Orange Pi использует `compose.orange-pi.yaml` и три изолированных контейнера:

| Контейнер | Назначение | Публикация |
|---|---|---|
| `filestorage-frontend` | nginx со статическим React-приложением и proxy `/api` | только `TAILSCALE_IP:3001` |
| `filestorage-backend` | Spring Boot API | только внутренняя Docker-сеть, порт `8081` |
| `filestorage-db` | PostgreSQL 17 | только внутренняя Docker-сеть, порт `5432` |

`filestorage-network` и том `filestorage-database` принадлежат только этому проекту. Файлы подключаются к backend через bind mount `UPLOADS_PATH`.

Внешний маршрут выглядит так:

```text
браузер
  -> HTTPS files.tm-port.ru
  -> nginx на VPS
  -> Tailscale 100.x.x.x:3001
  -> frontend nginx на Orange Pi
  -> /api -> backend -> PostgreSQL или UPLOADS_PATH
```

На домашнем роутере порты не открываются. Backend и PostgreSQL не доступны ни из интернета, ни напрямую через Tailscale. После выпуска TLS-сертификата приложение запускается с `APP_PUBLIC_URL=https://files.tm-port.ru` и `SECURE_COOKIE=true`.

Автозапуск управляется `/etc/systemd/system/filestorage.service`. Сервис запускается после `docker.service` и `tailscale-online.target`, дополнительно выполняет `tailscale wait`, затем принудительно пересоздаёт frontend. Это необходимо, потому что Docker не повторяет привязку порта к Tailscale IP после ошибки `cannot assign requested address`. База и backend при этом не пересоздаются.

`compose.prod.yaml` сохранён как альтернативная схема для сервера с уже существующими PostgreSQL `postgres` и внешней сетью `app-network`.

Полная инструкция и эксплуатационные команды приведены в [DEPLOYMENT_ORANGE_PI.md](DEPLOYMENT_ORANGE_PI.md).
